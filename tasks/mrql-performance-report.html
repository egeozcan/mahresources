MRQL Performance Improvement Roadmap

A sequenced implementation report for improving MRQL from low-risk measurement and indexing through set-based grouping, batched rendering, and optional specialized read models.

8 ordered workstreams SQLite + PostgreSQL Auth scope must remain fail-closed Target: million-resource deployments

Executive summary

The dominant opportunities are not in the MRQL parser. They are in generated SQL, pagination, repeated bucket execution, wide model materialization, and nested template queries. The highest-confidence sequence is to establish measurements, take low-risk index and projection wins, fix deep pagination, then replace bucketed GROUP BY's one-query-per-bucket algorithm.

The most important confirmed hotspot is executeBucketedQuery: it performs key discovery and then executes a full entity query per bucket, with a safety ceiling of 200 bucket queries and 10,000 materialized items. The most important architectural opportunity is therefore a set-based window query that preserves the existing limits and continuation semantics.

Current-state snapshot

Existing safeguards to preserve: default limits, interactive/export caps, query timeout and cancellation, forced principal scope, page query budgets, request-local result caching, bucket warnings, and continuation behavior.

Recommended order

  1. Benchmark and expose query counts/plans
  2. Audit indexes and reduce hydration/model width
  3. Add stable ordering and cursor pagination
  4. Rewrite bucketed grouping as a window query
  5. Batch nested template query patterns
  6. Optimize cross-entity top-N execution
  7. Add targeted hierarchy or metadata read models
  8. Evaluate an external read/search store

1 Benchmark and expose query counts/plans Create the evidence base needed for every subsequent optimization. Essential payoffLow complexityNo semantic change

Goal

Establish reproducible MRQL performance baselines for both supported databases. A change should not be considered a performance improvement until it is demonstrated against representative data distributions and query shapes.

Implementation details

  • Create deterministic fixture generators for approximately 100k, 1m, and multi-million entity datasets, with realistic relation density, hierarchy depth, duplicate names, metadata cardinality, and similarity rows.
  • Build benchmark cases for scalar filters, relation filters, subtree scope, hierarchy traversal, JSON metadata, FTS, similarity, flat pagination, aggregates, bucketed grouping, cross-entity sorting, and nested [mrql].
  • Record parse/bind/validate time separately from translation, SQL execution, model decoding, hydration, and template rendering.
  • Count actual SQL statements, returned rows, result bytes, cache hits/misses, count queries, and bucket fan-out.
  • Extend the explain surface to optionally run database-native EXPLAIN. The current endpoint returns generated SQL through GORM DryRun, not the optimizer's plan.
  • Fingerprint normalized queries so p50/p95/p99 can be grouped by shape without logging sensitive bound values.

Acceptance criteria

  • Benchmarks run reproducibly for SQLite and PostgreSQL.
  • Each benchmark reports wall time, SQL count, rows, allocations or result bytes, and query plan.
  • A stored baseline exists for the current implementation.
  • Performance regressions can be detected without relying on developer intuition.
  • Instrumentation does not expose query parameter secrets in logs.

2 Audit indexes and reduce unnecessary hydration/model width Take predictable, low-risk wins before changing query semantics or interfaces. High likely payoffLow–medium complexity

Goal

Make common relation filters and top-N reads use appropriate indexes, while reducing bytes read and decoded for result formats that do not need complete entities or rendering carriers.

Index audit

  • Inspect actual SQLite and PostgreSQL catalogs before adding migrations; GORM-generated join tables may already contain composite keys that are not obvious from model tags.
  • Verify both traversal orientations for tag, resource-note, group-resource, group-note, and group-group junction tables.
  • Prefer composites such as (entity_id, related_id) and (related_id, entity_id) where both directions are queried.
  • Check whether the PostgreSQL hash index on groups_related_resources(resource_id) receives scans. Remove it if the btree covers the workload.
  • Use measured workload data to evaluate top-N composites such as (owner_id, created_at DESC, id) and (resource_category_id, created_at DESC, id).
  • Evaluate expression or persisted effective-distance indexes for COALESCE(p_distance, hamming_distance).

Projection and hydration work

  • Define the minimum fields needed for raw IDs, compact lists, tables, and custom rendering.
  • Use narrow projections for count, existence, compact, and top-N candidate queries instead of scanning complete GORM models.
  • Skip category/type/custom-CSS/scope ancestry hydration when the requested format cannot consume it.
  • When hydration is needed, select only template and carrier fields required by rendering rather than entire rows.
  • Shorten the render-cache critical section: do not hold its mutex across slow database I/O. Reserve missing IDs, load outside the lock, and merge safely.

Important tradeoffs

  • Every added index consumes storage and slows relation writes, imports, and deletes.
  • Projection DTOs must not accidentally omit fields available to custom templates.
  • PostgreSQL trigram indexes are useful for proven substring workloads but can be large and write-expensive.
  • Arbitrary JSON keys should not receive blanket expression indexes; index only measured hot keys.

Acceptance criteria

  • Catalog inspection documents existing indexes on every MRQL junction table.
  • New indexes demonstrate improved plans on representative data.
  • Redundant indexes are identified, with safe removal handled separately if appropriate.
  • Compact/table paths avoid full render hydration where possible.
  • Response semantics and custom-template behavior remain unchanged.
  • Import and write benchmarks quantify index write amplification.

3 Add stable ordering and cursor/keyset pagination Prevent deep-page scans while retaining offset compatibility. Very high deep-page payoffMedium complexity

Goal

Provide an efficient continuation mechanism for large result sets. Existing page requests are converted into SQL OFFSET, which forces databases to find and discard preceding rows and can drift when ordering values tie.

Implementation details

  • Always append entity ID as the final tie-breaker for deterministic non-random ordering.
  • Define an opaque continuation token containing the effective order values, ID, entity type where needed, direction, and a version marker.
  • Translate the cursor to lexicographic predicates. For example, descending creation order becomes created_at < last_created OR (created_at = last_created AND id < last_id).
  • Handle null ordering explicitly and identically on SQLite and PostgreSQL.
  • Retain existing page/offset behavior for compatibility; add cursor mode alongside it.
  • Reject or specialize cursors for unstable orderings such as RANDOM().
  • Include effective query shape in the signed or validated cursor so it cannot be reused against an incompatible query.

Interface questions

Decision needed: whether cursor pagination is exposed only through JSON/CLI first or also replaces interactive list-page navigation. Starting as an opt-in JSON capability minimizes compatibility risk.

Verification

  • Differential tests should compare cursor and offset pages on an unchanged dataset.
  • Mutation tests should document expected behavior when rows are inserted or removed between page requests.
  • Benchmark first, middle, and very deep pages.
  • Cover ascending/descending fields, duplicate values, nulls, scoped principals, and all three entity types.

Acceptance criteria

  • Deterministic queries include a unique final ordering key.
  • Cursor pages do not use deep SQL offsets.
  • Existing offset clients continue to work.
  • Tokens fail safely when malformed or applied to another query shape.
  • SQLite and PostgreSQL return equivalent page sequences.

4 Rewrite bucketed grouping as a set-based window query Replace the confirmed one-query-per-bucket hotspot. Largest likely payoffHigh complexitySemantic risk

Goal

Reduce bucketed grouping from one discovery query plus up to 200 repeated item queries to one or a small constant number of SQL statements.

Proposed physical shape

WITH filtered AS (
  SELECT entity columns, bucket columns
  FROM ...
  WHERE original_predicates AND forced_scope
), ranked AS (
  SELECT *,
         ROW_NUMBER() OVER (
           PARTITION BY bucket_columns
           ORDER BY requested_order, id
         ) AS bucket_row
  FROM filtered
)
SELECT *
FROM ranked
WHERE bucket_row <= per_bucket_limit
ORDER BY bucket_columns, bucket_row;

The implementation may still use a second narrow statement for bucket continuation or totals, but it should not execute one statement per bucket.

Required semantic preservation

  • Relation buckets must retain identity keys even when display names collide.
  • Null bucket keys and multi-valued relations must match current behavior.
  • Each emitted bucket receives its full per-bucket limit; the global cap must not cut a bucket midway and make remaining items unreachable.
  • MaxBuckets, maxBucketedTotalItems, warnings, TotalGroups, and NextOffset remain meaningful.
  • Forced principal scope must be part of the filtered relation before ranking.
  • Duplicate elimination caused by many-to-many joins must remain correct.

Delivery strategy

  1. Capture the existing behavior with a large differential test matrix.
  2. Add an internal execution strategy seam without changing callers.
  3. Implement the window strategy for simple scalar buckets first.
  4. Expand to relation and computed/date buckets.
  5. Run old and new strategies against identical fixtures and compare normalized results.
  6. Remove or retain the old implementation as a temporary fallback only after both databases pass parity and performance gates.

Acceptance criteria

  • Bucketed execution uses a constant number of SQL statements relative to bucket count.
  • Old/new differential tests cover nulls, relations, ordering, limits, caps, continuation, and scoped users.
  • Both databases show a substantial win on high-bucket benchmarks.
  • No bucket is silently truncated in a way that makes items unreachable.
  • Timeout, cancellation, and explain diagnostics still identify the execution phase.

5 Batch common nested-template MRQL patterns Eliminate per-card query fan-out incrementally before building a general render planner. High dashboard payoffMedium–high complexity

Goal

Convert common entity-correlated queries from one execution per rendered card into one grouped execution per query pattern.

Candidate patterns

  • Count resources, notes, or child groups for every displayed group.
  • Fetch the newest N owned entities for every displayed group.
  • Resolve the same saved query with only the scope/entity parameter changing.
  • Compute tag or category counts across the full displayed entity set.

Incremental design

  1. Instrument query templates and identify the highest-frequency correlated shapes.
  2. Canonicalize the query with the varying entity/scope value represented as a batch dimension.
  3. Collect all dimension IDs before rendering cards.
  4. Execute one grouped query and distribute results through a request-local data-loader map.
  5. Fall back to the existing executor for patterns that cannot be safely batched.

Supporting cache improvements

  • Move request-local result caching below saved-query resolution and scope binding.
  • Use canonical AST-based keys rather than raw query strings.
  • Coalesce in-flight identical calls so concurrent rendering cannot stampede.
  • Bound entries, rows, and approximate bytes when the page query budget is disabled.
  • Count {total}'s second SQL query explicitly in diagnostics and budgets.

Acceptance criteria

  • A representative N-card dashboard executes O(1) or a small constant number of SQL statements for supported patterns.
  • Unsupported query forms preserve current behavior.
  • Cache keys include resolved scope and execution policy, preventing authorization leakage.
  • Rendered output matches the non-batched executor.
  • Query budget reporting reflects actual SQL work.

Future extension

If several narrow batchers prove valuable, generalize them into a render planner: discover dependencies, plan compatible batches, execute, hydrate, and only then render. Do not begin with a universal planner before measured patterns justify it.

6 Optimize cross-entity top-N execution Move global candidate selection into SQL and hydrate only selected entities. High for cross-entity useHigh complexity

Goal

Avoid fetching up to offset + limit complete rows from each of resources, notes, and groups, then globally sorting them in Go.

Proposed design

  1. Translate each applicable entity branch to a narrow projection: entity type, ID, and common sort fields.
  2. Combine branches with UNION ALL.
  3. Apply global deterministic ordering and cursor/limit in SQL.
  4. Batch-fetch complete entities only for the selected IDs.
  5. Reassemble output in the global selected order.

Challenges

  • Only fields common across entity types can participate in global ordering without a defined coercion.
  • Entity-specific predicates must continue to skip inapplicable branches safely.
  • Random cross-entity sampling currently contains population-proportional logic and may require a separate physical strategy.
  • Timeout behavior currently tolerates per-entity timeouts as warnings; a unified query changes that failure model.
  • Result hydration must preserve global ordering after typed rows are fetched.

Acceptance criteria

  • Global top-N selection occurs before full-model hydration.
  • Deep continuation does not materialize all preceding candidates.
  • Supported sort orders match current results with ID tie-breaking.
  • Scoped authorization is enforced independently in every union branch.
  • Random semantics are either preserved through a specialized strategy or explicitly excluded.

Relevant code

application_context/mrql_context.go:732+, mrql/translator.go, cross-entity sorting/random/count tests

7 Add targeted hierarchy or metadata read models Use denormalization only where measured recursive or JSON workloads justify it. Potentially high payoffHigh migration costWrite consistency risk

Decision gate

This task should begin only if benchmark and production evidence shows hierarchy traversal or arbitrary metadata to be a sustained bottleneck after earlier work. These are separate optional modules and need not ship together.

Option A: hierarchy closure or materialized path

Maintain ancestry data such as (ancestor_id, descendant_id, depth), or a materialized path/root/depth representation. It could serve MRQL scope, ancestor/descendant predicates, root resolution, and render ancestry.

  • Design group moves as transactional updates of hierarchy data.
  • Backfill existing trees and detect cycles or depth violations.
  • Keep authorization fail-closed if hierarchy projection is stale or inconsistent.
  • Benchmark read improvements against move and import write amplification.

Option B: indexed metadata projection

Maintain a typed read model such as:

(entity_type, entity_id, key, text_value, numeric_value, date_value)

Candidate indexes include (entity_type, key, text_value, entity_id) and numeric/date equivalents.

  • Update the projection transactionally or through a reliable queue with reconciliation.
  • Define type coercion exactly so projected comparisons match existing JSON semantics.
  • Continue using direct JSON extraction for keys not present in the projection.
  • Provide consistency checks and rebuild tooling.

Acceptance criteria

  • A measured workload justifies the additional read model.
  • Backfill and rebuild procedures are documented and tested.
  • Writes cannot leave authorization hierarchy silently stale.
  • Projection consistency is observable and repairable.
  • Query semantics are differential-tested against the canonical tables/JSON.

8 Evaluate an external read/search store Reserve specialized infrastructure for search, faceting, or analytics workloads the primary databases cannot satisfy economically. Potentially transformativeVery high complexityOperational risk

Goal

Determine whether an optional search or reporting adapter—such as OpenSearch, Meilisearch, or a PostgreSQL reporting schema—is warranted for richer ranking, faceting, typo tolerance, or large analytical workloads.

Preconditions

  • Native PostgreSQL FTS/trigrams or SQLite FTS5 have been measured and found insufficient.
  • The target workload is clearly separated from transactional CRUD.
  • The product can tolerate defined eventual-consistency windows.
  • Operators accept another service, index lifecycle, monitoring, backups, and recovery procedures.

Safety architecture

  • The external store must not become the authoritative source of subtree authorization.
  • Either encode and update scope membership reliably in the index or re-check selected IDs against the primary database before returning results.
  • Use an outbox/indexing queue with replay and reconciliation rather than best-effort hooks.
  • Version index schemas and support zero-downtime rebuilds.
  • Define behavior when the external store is unavailable: fail clearly or fall back only for supported query classes.

Evaluation criteria

AreaQuestions
LatencyDoes it materially beat tuned native FTS and indexed SQL at target cardinality?
FreshnessWhat lag is acceptable for creates, edits, moves, and permission changes?
AuthorizationHow are group-subtree moves invalidated, and how is fail-closed behavior proven?
OperationsWho runs, upgrades, backs up, and monitors the service?
SemanticsWhich MRQL predicates can execute externally without changing results?

Acceptance criteria

  • A written benchmark demonstrates value over native alternatives.
  • The adapter is optional and the primary database remains authoritative.
  • Index lag, failures, and reconciliation are observable.
  • Authorization tests prove no out-of-scope IDs can leak.
  • Operational cost and deployment support are explicitly accepted.

Avoid premature optimization

  • Do not prioritize parser caching over measured SQL and rendering costs.
  • Do not add every possible composite or JSON expression index.
  • Do not introduce a global result cache without complete invalidation and principal-scope keys.
  • Do not apply COUNT(*) OVER() universally; it may make selective limited queries scan more rows.
  • Do not replace recursive authorization scope with denormalized data until consistency and fail-closed behavior are proven.
  • Do not remove offset pagination abruptly; ship keyset continuation alongside it.
  • Do not make an external search engine authoritative for access control.

Program-level completion gates

  • All changes preserve forced principal subtree scoping.
  • SQLite and PostgreSQL result equivalence is covered by integration tests.
  • Interactive and export policies remain independently bounded.
  • Request cancellation and runtime query timeout remain effective.
  • Performance claims include before/after data on representative cardinalities.
  • Write and import regressions from new indexes or projections are measured.
  • The full Go, browser E2E, CLI E2E, and PostgreSQL suites pass for implementation phases.

Basis: static review of the current MRQL translator, execution, rendering, cache, hierarchy, and index paths. Exact planner behavior and expected gains must be validated against representative production-like datasets.
