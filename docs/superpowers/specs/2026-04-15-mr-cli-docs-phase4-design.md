# mr CLI Documentation Overhaul — Phase 4 Design (Delta)

**Date:** 2026-04-15
**Status:** Approved for planning
**Parent spec:** `docs/superpowers/specs/2026-04-14-mr-cli-docs-design.md`
**Sibling spec:** `docs/superpowers/specs/2026-04-15-mr-cli-docs-phase3-design.md`
**Scope:** Execution-level design for Phase 4 (polish + cleanup). Closes the mr CLI documentation overhaul by deleting the superseded hand-written CLI page, wiring the generated `docs-site/docs/cli/` tree into the Docusaurus sidebar, updating project-level references (README, `CLAUDE.md`, stray internal links), shipping two previously-optional quality fixes (`note-type get --json` projection widening and a standalone MRQL reference page), strengthening one weak doctest assertion (`mr search`) surfaced during spec review, and running a full verification sweep.

## Relationship to the Parent and Sibling Specs

The parent spec (2026-04-14) defines the template, the `helptext` package, the `mr docs` command group, the doctest runner, and the docs-site generation pipeline. The Phase 3 delta (2026-04-15) covered migration of the remaining 147 commands and wired the `cli-docs-fresh` and `cli-doctest` CI jobs. Neither is changed here.

Phase 4 is strictly additive cleanup. It ships eight work items across six commits on `master`. No new design primitives are introduced.

### 2. Wire the generated CLI tree into the Docusaurus sidebar

`docs-site/docs/cli/` currently exists on disk but is not referenced by `docs-site/sidebars.ts`. Deleting `features/cli` without adding a replacement would remove the only navigation path into the CLI documentation.

Replace the sidebar's `features/cli` entry (Advanced Features, line 79) with a new top-level section between Advanced Features and Guides:

```ts
{
  type: 'category',
  label: 'CLI Reference',
  link: { type: 'doc', id: 'cli/index' },
  items: [{ type: 'autogenerated', dirName: 'cli' }],
},
```

Rationale for `autogenerated`: the `cli/` tree is regenerated on every CLI help-text change by `npm run docs-gen`, and `cli-docs-fresh` CI enforces that commit-time consistency. An autogenerated sidebar entry means new commands added to the CLI automatically appear in the nav with zero maintenance — matching the "one source of truth" principle the overhaul is built on.

Rationale for `link: { type: 'doc', id: 'cli/index' }`: the `cli/index.md` landing page already contains a searchable table of every command. Attaching it as the category header makes clicking "CLI Reference" in the nav navigate to the landing page instead of just expanding the category.

### 6. Widen the `note-type get --json` projection

`cmd/mr/commands/note_types.go` currently projects the server's note-type API response into a 5-field `noteTypeResponse` struct (`ID`, `Name`, `Color`, `SchemaID`, `CreatedAt`) for both the table and `--json` output paths. This is asymmetric with `mr resource get`, which passes the raw response body through for `--json` and uses a narrow projection only for the table.

Fix: follow the `resource get` pattern. Keep the narrow projection for the table path; for the `--json` path, unmarshal the response into a `json.RawMessage` (or equivalent passthrough) and print it directly. The server response shape determines what fields surface; the CLI no longer drops them.

Verification:

- Update any doctest in `note_types_help/note_type_get.md` whose assertion depends on the pre-widening shape. Likely an assertion on `.ID` which survives; if an assertion on a projected field like `.Color` exists it stays correct since the widened output is a superset.
- Regenerate `docs-site/docs/cli/note-type/get.md` via `npm run docs-gen`.
- No server-side changes. This is pure CLI-side output handling.

### 7. Extract the MRQL cheatsheet to a standalone docs-site page

Create `docs-site/docs/features/mrql-reference.md` holding the full DSL reference: syntax, operators, supported fields, scope keywords, and worked examples. Source material: the pre-Phase-3 MRQL Long (recoverable from `git log -p cmd/mr/commands/mrql_help/mrql.md`) plus any cheatsheet fragments in `docs-site/docs/features/mrql.md` that belong with the reference rather than the conceptual overview.

Wiring:

- `docs-site/docs/features/mrql.md` — add a "See also" line at the end pointing to `./mrql-reference`.
- `docs-site/sidebars.ts` — add `'features/mrql-reference'` immediately after the existing `'features/mrql'` entry in the Advanced Features items array so the two live together in nav.

### 8. Strengthen the `mr search` doctest assertion

Surfaced during spec review. The current doctest in `cmd/mr/commands/search_help/search.md:24–27` creates a uniquely-named group and then asserts only that the response has a valid shape:

```bash
# mr-doctest: create a uniquely-named group and confirm search returns a response shape
NAME="doctest-search-$$-$RANDOM"
mr group create --name "$NAME" --json > /dev/null
mr search "$NAME" --json | jq -e '.total >= 0 and (.results | type == "array")'
```

That assertion passes against a completely broken search index (which would still emit `{"total": 0, "results": []}` for any query). Tighten it to verify that the freshly-created group actually surfaces in results. Prefer asserting by the created group's numeric ID rather than by name, to avoid dependency on FTS tokenization quirks (dashes, underscores, digit-only suffixes):

```bash
# mr-doctest: create a uniquely-named group and confirm search finds it
NAME="doctest-search-$$-$RANDOM"
GID=$(mr group create --name "$NAME" --json | jq -r '.ID')
mr search "$NAME" --json | jq -e --argjson g "$GID" '.total >= 1 and ([.results[].id] | any(. == $g))'
```

Plan-time verification:

- Confirm FTS indexing is synchronous on `POST /v1/group` (SQLite FTS5 triggers typically are; if there's an async path, the assertion will flake). If async, fall back to a short retry loop or polling inside the doctest.
- Regenerate `docs-site/docs/cli/search.md` via `npm run docs-gen`.
- Run `./mr docs check-examples --server http://127.0.0.1:8192 --environment ephemeral` with just this block and confirm PASS before committing.

Out of scope for this item: auditing every other doctest for similarly weak shape-only assertions. The `mr search` case was surfaced by review; a systematic audit belongs in a Phase 5 ticket.

## Test & Verification Plan

### Per-commit Local Simulation

Before each of the five commits, run:

```bash
go build -o mr ./cmd/mr/
./mr docs lint                                                          # mirrors cli-docs-fresh's lint step
npm run build-cli && npm run docs-gen                                   # regenerate if help text changed
git diff --exit-code -- docs-site/docs/cli/                             # mirrors cli-docs-fresh's fresh check
```

For commits that touch CLI help text (#6, #7) additionally run:

```bash
ls mahresources || go build --tags 'json1 fts5' -o mahresources         # per gotcha: CLI-only build doesn't produce server
./mahresources -ephemeral -bind-address=:8192 -max-db-connections=2 &
SERVER_PID=$!; sleep 1
./mr docs check-examples --server http://127.0.0.1:8192 --environment ephemeral
kill $SERVER_PID; wait $SERVER_PID 2>/dev/null
```

Port `8192` not `8181` — the developer dev server holds `8181`.

### Full Test Sweep After All Commits Land

Run in this exact order. Each step must be green before proceeding.

1. **Binary freshness:** `ls mahresources` → if missing, `go build --tags 'json1 fts5' -o mahresources`. `go build -o mr ./cmd/mr/`.
2. **Go unit tests:** `go test --tags 'json1 fts5' ./...`. Expected: all pass, including `TestLintRealTree` and `cmd/mr/helptext/` parser tests.
3. **Lint gate:** `./mr docs lint`. Expected: `OK: 3 warnings` (the 3 `docs *` commands that do not doctest themselves; this is Phase 3 steady-state).
4. **Doctest gate:** start an ephemeral server on `:8192` (see command block above), run `./mr docs check-examples --server http://127.0.0.1:8192 --environment ephemeral`. Expected: exit `0`, `159 PASS + 2 SKIP` (Phase 3 steady-state; skips are `resource from-url` and `resource from-local`).
5. **CLI doctest E2E:** `cd e2e && npm run test:with-server:cli-doctest`. Expected: Playwright green.
6. **CLI E2E (parallel):** `cd e2e && npm run test:with-server:cli`. Expected: 304 tests green. If the job stalls (gotcha #3), raise `-max-db-connections=2` in `e2e/scripts/run-tests.js` as a first-try mitigation; if still flaky, document as a known flake and move on — the serial `cli-doctest` job is the authoritative CI gate.
7. **Postgres tests:** `go test --tags 'json1 fts5 postgres' ./mrql/... ./server/api_tests/... -count=1`. Requires Docker. Expected: all pass.

## Risks and Mitigations

**Risk: First real exercise of `cli-docs-fresh` / `cli-doctest` CI jobs fails on push.**
Mitigation: per-commit local simulation that mirrors each CI job's steps exactly. Ordering commits by CI surface (commit 1 has zero CLI/docs-site-tree surface) means commit 1 verifies the CI wiring itself. Recovery: fix forward on master; each commit is small enough to debug in isolation.

**Risk: Sidebar `autogenerated` picks up files that shouldn't be navigable.**
Mitigation: the `cli/` tree is 100% generated from CLI help; every file there is intentional. Visually verify in a local `cd docs-site && npm run start` before pushing commit 2.

**Risk: `cli/index.md`'s `sidebar_label: CLI` conflicts with the category label `CLI Reference`.**
Mitigation: `link: { type: 'doc', id: 'cli/index' }` attaches the doc to the category header itself, not as a sibling item. The category label comes from `label: 'CLI Reference'` in the sidebar config; the doc's own `sidebar_label` does not appear as a nav entry when the doc is a category link. Verify visually on local Docusaurus before pushing.

**Risk: `npm run test:with-server:cli` flake (gotcha #3 from Phase 3).**
Mitigation: if it reproduces in step 6 of the full test sweep, raise `-max-db-connections=2` in `e2e/scripts/run-tests.js` as first-try fix; if still flaky, document in this spec's addendum and move on. The serial `cli-doctest` CI job is the authoritative gate on CLI behavior.

**Risk: Widening `note-type get --json` breaks the non-JSON table output.**
Mitigation: the change is scoped to the `--json` branch of the command's `RunE`. The table-path projection is untouched. Doctests and E2E CLI specs for `mr note-type get` (without `--json`) must still pass.

**Risk: Stronger `mr search` assertion flakes against an async FTS indexer.**
Mitigation: verify during plan execution whether `POST /v1/group` updates the FTS index synchronously (inspect the server path in `application_context/group_context.go` and the DB triggers). If async, add a short bounded retry loop inside the doctest (try up to 3 times with a small sleep) rather than weakening the assertion back to shape-only. If retry also proves flaky, fall back to asserting `.total >= 1` by name match (less ID-robust but still stronger than shape-only) and record the trade-off inline in the doctest comment.

**Risk: MRQL reference page creates circular nav confusion between `features/mrql` (concept) and `features/mrql-reference` (reference).**
Mitigation: the two are adjacent in the sidebar and cross-linked. Clear labels: "MRQL" for the concept page, "MRQL Reference" for the DSL reference. Prose at the top of each page explicitly directs readers to the other for the opposite use case.

**Risk: Phase 4 scope creep during execution.**
Mitigation: out-of-scope items surfaced during the test sweep (cosmetic docs drift, flaky unrelated tests, newly-discovered bugs unrelated to CLI docs) get captured as follow-ups. Phase 4 does not expand to absorb them.
